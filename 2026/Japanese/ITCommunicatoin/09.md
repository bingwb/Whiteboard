在项目开发中，Git 的操作不仅仅是敲命令，还涉及到与同事的沟通。日本现场通常非常看重 「ブランチ運用ルール（分支管理规则）」，因此在使用命令时，往往伴随着特定的日语描述。
以下是为您整理的 Git <ruby>常用<rt>じょうよう</rt></ruby>コマンド・<ruby>用語集<rt>ようごしゅう</rt></ruby>。

🌿 Git <ruby>常用<rt>じょうよう</rt></ruby>コマンド & <ruby>現場<rt>げんば</rt></ruby>フレーズ

1. <ruby>基本<rt>きほん</rt></ruby>コマンド - 基础操作

| 命令 | 日语描述 | 场景说明 |
|---|---|---|
| git clone | クローン | 将远程仓库复制到本地。 |
| git status | <ruby>状態確認<rt>じょうたいかくにん</rt></ruby> | 查看当前改动的文件。 |
| git add | ステージング | 将修改加入暂存区。 |
| git commit | コミット | 提交更改。现场常说「細（こま）かくコミットする」。 |
| git push | プッシュ | 将本地提交推送到服务器。 |
| git pull | プル | 获取并合并远程更改。 |
| git fetch | フェッチ | 仅获取远程更新，不合并（用于确认差异）。 |

2. ブランチ<ruby>操作<rt>そうさ</rt></ruby> - 分支管理
日本现场非常强调“不要在本地堆积改动”，要勤切分支。
 * git branch / git checkout -b
   * <ruby>枝分<rt>えだわ</rt></ruby>かれ / ブランチを<ruby>切<rt>き</rt></ruby>る: 切分支。
   * 例句： 「<ruby>新機能<rt>しんきのう</rt></ruby>のブランチを**<ruby>切<rt>き</rt></ruby>りました**。」 (切好了新功能的分支。)
 * git merge
   * マージ: 合并。
   * 例句： 「developブランチをfeatureブランチにマージしました。」
 * git rebase
   * リベース: 变基。日本很多团队喜欢用 rebase 来保持提交线性的整洁（避免过多的 Merge commit）。
 * git checkout / git switch
   * <ruby>切<rt>き</rt></ruby>り<ruby>替<rt>か</rt></ruby>える: 切换分支。

3. <ruby>応用<rt>おうよう</rt></ruby>・<ruby>修正<rt>しゅうせい</rt></ruby> - 进阶与修正
当代码写乱了或者需要撤销时：
 * git stash
   * <ruby>一時退避<rt>いちじたいひ</rt></ruby>: 临时退避/缓存。当你想切分支但当前改动还没提交时必用。
   * 例句： 「<ruby>作業中<rt>さぎょうちゅう</rt></ruby>のコードを一旦（いったん）スタッシュしました。」
 * git cherry-pick
   * チェリーピック: 摘取特定的 commit。常用于将某个 Bug 修复应用到其他分支。
 * git reset --hard
   * <ruby>強制戻<rt>きょうせいもど</rt></ruby>し / リセット: 强制回滚。慎用！
 * git revert
   * <ruby>打<rt>う</rt></ruby>ち<ruby>消<rt>け</rt></ruby>し: 撤销提交。通过创建一个新的反向提交来抵消之前的错误。
 * git diff
   * <ruby>差分確認<rt>さぶんかくにん</rt></ruby>: 查看差异。

4. <ruby>現場<rt>げんば</rt></ruby>でよく<ruby>起<rt>お</rt></ruby>こるトラブル - 常见问题处理
 * コンフリクト (Conflict): 冲突。
   * 例句： 「マージしたらコンフリクトが<ruby>発生<rt>はっせい</rt></ruby>したので、<ruby>解消<rt>かいしょう</rt></ruby>します。」 (发生冲突了，我去解决一下。)
 * <ruby>先祖返<rt>せんぞがえ</rt></ruby>り (Regression): 之前修好的代码又被旧代码覆盖了。
   * 这在日本现场是严重事故，通常是因为不小心强制 Push 或错误的 Merge 导致的。

5. <ruby>運用<rt>うんよう</rt></ruby>ルール - 运作规则常见表达
 * プルリクエスト (PR) / マージリクエスト (MR): 合并请求。
 * レビュー<ruby>依頼<rt>いらい</rt></ruby>: 请求评审。
 * コンフリクト<ruby>解消<rt>かいしょう</rt></ruby>: 解决冲突。
 * <ruby>最新<rt>さいしん</rt></ruby>を<ruby>取<rt>と</rt></ruby>り<ruby>込<rt>こ</rt></ruby>む: 拉取最新代码。
   * 例句： 「developの**<ruby>最新<rt>さいしん</rt></ruby>を<ruby>取<rt>と</rt></ruby>り<ruby>込<rt>こ</rt></ruby>んで**からプッシュしてください。」 (请先合并 develop 最新的代码再提交。)
💡 建议的 Git Commit Message 规范 (日本现场习惯)
虽然很多公司有自己的规则，但常见的格式如下：
 * [add] 〇〇機能の追加 (添加了某功能)
 * [fix] 〇〇バグの修正 (修复了某 Bug)
 * [update] 〇〇定義の更新 (更新了定义)
 * [refactor] コードの共通化 (代码提取/重构)
