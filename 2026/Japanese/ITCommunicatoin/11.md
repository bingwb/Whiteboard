在 Azure Repos 的开发流程中，PR 评审（Code Review） 往往是进度最容易卡住的地方。
如果一直等不到审核，或者需要解释复杂的逻辑，直接催促会显得无礼。以下是针对 PR 催办、逻辑解释以及应对刁钻评审的委婉日语模板。

🚀 Azure Repos <ruby>レビュー<rt>れびゅー</rt></ruby><ruby>促進<rt>そくしん</rt></ruby>・<ruby>交渉<rt>こうしょう</rt></ruby>フレーズ
1. 催促评审（<ruby>催促<rt>さいそく</rt></ruby>）
> 当 PR 提交了一两天还没动静，而你又急着合并时。
> 
 * お<ruby>忙<rt>いそが</rt></ruby>しいところ<ruby>恐縮<rt>きょうしゅく</rt></ruby>ですが、こちらのPRを<ruby>確認<rt>かくにん</rt></ruby>いただけますでしょうか。
   * (百忙之中深感抱歉，能请您确认一下这个 PR 吗？)
 * こちらの<ruby>修正<rt>しゅうせい</rt></ruby>ですが、<ruby>後続<rt>こうぞく</rt></ruby>タスクの<ruby>影響<rt>えいきょう</rt></ruby>があるため、<ruby>早<rt>はや</rt></ruby>めにレビューをいただけると<ruby>助<rt>たす</rt></ruby>かります。
   * (关于这次修正，因为会影响后续任务，如果您能早点评审就太感谢了。)
 * お<ruby>手空<rt>てあ</rt></ruby>きの<ruby>際<rt>さい</rt></ruby>で<ruby>構<rt>かま</rt></ruby>いませんので、ご確認（かくにん）をお<ruby>願<rt>ねが</rt></ruby>いします。
   * (在您方便的时候确认即可，拜托了。) —— 虽然说是“方便时”，但在日本语境下这就是在提醒对方去看。

2. 解释复杂的逻辑（<ruby>ロジック<rt>ろじっく</rt></ruby>の<ruby>説明<rt>せつめい</rt></ruby>）
> 当改动很大，担心 Reviewer 看不懂或产生误解时。
> 
 * <ruby>変更箇所<rt>へんこうかしょ</rt></ruby>が<ruby>多<rt>おお</rt></ruby>いため、<ruby>概要<rt>がいよう</rt></ruby>をコメントに<ruby>追記<rt>ついき</rt></ruby>しました。
   * (由于修改的地方较多，我在评论里补充了概要。)
 * 〇〇の<ruby>不具合<rt>ふぐあい</rt></ruby>を<ruby>修正<rt>しゅうせい</rt></ruby>するため、あえてこのような<ruby>実装<rt>じっそう</rt></ruby>にしています。
   * (为了修复 XX 的 Bug，特意采用了这种实现方式。)
 * <ruby>既存<rt>きぞん</rt></ruby>への<ruby>影響<rt>えいきょう</rt></ruby>を<ruby>最小限<rt>さいしょうげん</rt></ruby>にするため、この<ruby>方針<rt>ほうしん</rt></ruby>を<ruby>採用<rt>さいよう</rt></ruby>しました。
   * (为了将对现有系统的影响降到最低，我采用了这个方针。)

3. 委婉拒绝评审意见（<ruby>指摘<rt>してき</rt></ruby>への<ruby>反論<rt>はんろん</rt></ruby>）
> 对方提的意见不合理，或者按他的改法会出问题时。
> 
 * ご<ruby>指摘<rt>してき</rt></ruby>の<ruby>通<rt>とお</rt></ruby>りですが、〇〇のケースを<ruby>考慮<rt>こうりょ</rt></ruby>すると、<ruby>現状<rt>げんじょう</rt></ruby>のままでよいかと<ruby>考<rt>かんが</rt></ruby>えています。
   * (您说得对，但考虑到 XX 的情况，我觉得维持现状可能更好。)
 * その<ruby>修正<rt>しゅうせい</rt></ruby>を<ruby>行<rt>おこな</rt></ruby>うと、△△でエラーが<ruby>発生<rt>はっせい</rt></ruby>する<ruby>可能性<rt>かのうせい</rt></ruby>があります。
   * (如果进行那项修改的话，△△ 可能会发生错误。)
 * <ruby>別途<rt>べっと</rt></ruby>、<ruby>対面<rt>たいめん</rt></ruby>（またはTeams）で<ruby>詳細<rt>しょうさい</rt></ruby>を<ruby>説明<rt>せつめい</rt></ruby>させていただいてもよろしいでしょうか？
   * (可以让我另外当面或在 Teams 上为您详细解释一下吗？) —— 打字说不清时，面对面沟通效率最高。

4. Azure Repos 界面操作的相关对话
 * 「ポリシーにより、Required Reviewer（<ruby>必須<rt>ひっす</rt></ruby>レビュワー）の<ruby>承認<rt>しょうにん</rt></ruby>が<ruby>必要<rt>ひつよう</rt></ruby>です。」
   * (受策略限制，需要强制评审人的通过。)
 * 「コンフリクトをWeb<ruby>上<rt>じょう</rt></ruby>で<ruby>解消<rt>かいしょう</rt></ruby>しました。」
   * (已在网页端解决了冲突。)
 * 「<ruby>修正<rt>しゅうせい</rt></ruby>が<ruby>終<rt>お</rt></ruby>わりましたので、Re-activate（<ruby>再<rt>さい</rt></ruby>アクティブ<ruby>化<rt>か</rt></ruby>）しました。」
   * (修正完了，我已经把 PR 重新激活了。)

💡 沟通小贴士：关于 Azure Repos 的评论状态
在 Azure Repos 中，评论有不同状态（Active, Resolved, Pending）。
 * 当你改完代码后，记得把 Reviewer 的评论标记为 Resolved。
 * 同时在评论下回复：「<ruby>対応<rt>たいおう</rt></ruby>しましたので、クローズします。」 (已经处理了，现在关闭此评论。)
这样做会让 Reviewer 觉得你工作非常细致，好感度倍增！
