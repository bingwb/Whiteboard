既然你正在使用 Azure Repos，你会发现它虽然也是 Git 仓库，但在界面和术语上与 GitHub/GitLab 有一些细微差别（比如 GitHub 叫 Pull Request，Azure Repos 也叫 Pull Request，但操作按钮和策略设置的位置不同）。
以下是针对 Azure Repos 环境整理的常用术语与对话笔记。

🌀 Azure Repos <ruby>専用<rt>せんよう</rt></ruby>・<ruby>現場<rt>げんば</rt></ruby>ノート
1. Azure Repos <ruby>特有<rt>とくゆう</rt></ruby>の<ruby>用語<rt>ようご</rt></ruby> - 专用术语

| 术语 | 读音 | 说明 |
|---|---|---|
| Pull Request (PR) | プルリク / ピーアール | 合并请求。日本现场最常叫「プルリク」。 |
| Reviewer | レビュワー | 审核人。通常分为 Required（强制）和 Optional（可选）。 |
| Branch Policy | ブランチポリシー | 分支策略。如：禁止直接 Push 到 main。 |
| Policies | ポリシー | 各种限制规则（如必须编译通过才能合并）。 |
| Conflict | コンフリクト | 冲突。Azure Repos 可以在 Web 端解决简单冲突。 |
| Complete | コンプリート | 完成/合并 PR。Azure Repos 的按钮是 "Complete"。 |
| Cherry-pick | チェリーピック | 摘取提交。Azure Repos 界面支持一键 Cherry-pick。 |

3. Azure Repos <ruby>特有<rt>特有</rt></ruby>の<ruby>機能<rt>きのう</rt></ruby>に関（かん）する<ruby>会話<rt>かいわ</rt></ruby>
Azure Repos 与 Azure Boards（任务卡片）结合非常紧密，经常会涉及到“关联”操作。
 * Work Itemの<ruby>紐付<rt>ひもづ</rt></ruby>け (关联工作项):
   * 「PRにワークアイテムを<ruby>紐付<rt>ひもづ</rt></ruby>けてください。」
   * (请在 PR 中关联工作项/Ticket。) —— 日本现场非常看重 Traceability（追溯性）。
 * Auto-complete (自动完成):
   * 「オートコンプリートをセットしておきました。」
   * (我已经设置了自动合并。) —— 当所有策略（如编译、审核）满足后自动合并。
 * Abandon (废弃):
   * 「このPRは<ruby>不要<rt>ふよう</rt></ruby>になったので、アバンドン（Abandon）します。」
   * (这个 PR 不需要了，我把它废弃掉。)

4. <ruby>重要<rt>じゅうよう</rt></ruby>：Azure Repos でのトラブル<ruby>対応<rt>たいおう</rt></ruby>
 * 「ブランチポリシーでプッシュが<ruby>拒否<rt>きょひ</rt></ruby>されました。」
   * (因为分支策略，Push 被拒绝了。) —— 通常是因为你想直接 Push 到受保护的分支。
 * 「ビルドパイプラインが<ruby>失敗<rt>しっぱい</rt></ruby>しているため、マージできません。」
   * (因为编译流水线失败，无法合并。)
 * 「リモートとコンフリクトしているので、ローカルでリベースします。」
   * (因为与远程冲突，我会在本地做 Rebase。)
💡 Azure Repos <ruby>運用<rt>うんよう</rt></ruby>のコツ (运作小贴士)
 * 关联 ID： 提交时在注释里加上 #123（Work Item ID），Azure Repos 会自动关联，这在写周报或追踪 Bug 时非常有用。
 * Squash Merge： 日本团队为了让 main 分支的历史好看，经常要求使用 Squash Merge（スカッシュマージ）。合并时记得确认勾选这个选项。
 * 描述规范： PR 的描述里通常要写 【<ruby>目的<rt>もくてき</rt></ruby>】、【<ruby>影響範囲<rt>えいきょうはんい</rt></ruby>】、【テスト<ruby>結果<rt>けっか</rt></ruby>】。
